Executive Summary

Overview of DFRuleProcessorService

This document describes the DFRuleProcessorService: an Azure Function that receives a pointer to transaction payloads (via Event Grid), evaluates a set of business rules against those payloads using a rules engine, and produces transformed output (modified and/or cloned transactions) which it persists to blob storage and signals to downstream consumers via a queue.

Key Concept: The service is deliberately hybrid — a declarative rules engine is used for simple transformations, while a C# escape hatch (custom actions) handles complex operations.

What This Service Does

  • Receives transaction payload notifications via Azure Event Grid
  • Applies tenant-specific business rules using a flexible rules engine
  • Transforms transactions through updates, cloning, and custom actions
  • Persists results to blob storage as immutable output artifacts
  • Notifies downstream systems via queue messages

Hybrid Approach

The service combines two powerful paradigms:

  • Declarative Rules: JSON-based workflow definitions for straightforward transformations
  • Procedural Actions: C# custom action handlers for complex business logic that requires procedural control

Goals and Scope

Primary objectives and document boundaries

Primary Goals

  • Apply tenant-specific business rules to incoming transaction payloads with flexibility and precision
  • Produce immutable output artifacts (new blobs) that represent the post-enrichment result, ensuring data integrity and traceability
  • Support multiple transformation patterns:
    • Direct property updates on existing transactions
    • Cloning transactions with simple property changes
    • Cloning with derived values computed at runtime
    • Full custom procedural actions for complex scenarios

Key Design Principles

  • Separation of Concerns: Clear boundaries between event handling, orchestration, rule evaluation, and persistence
  • Declarative First: Simple transformations use JSON rules; complex logic falls back to C# methods
  • Immutability: Original payloads remain unchanged; all transformations produce new output artifacts
  • Extensibility: Custom actions can be added without modifying core engine logic

End-to-End Flow

Step-by-step processing pipeline

Processing Steps

Step 1: Event Trigger

Event Grid fires with blob URL when a new transaction payload is uploaded to storage. The event contains metadata including the blob location and trigger timestamp.

Step 2: Function Invocation

AtlasFunction validates the incoming event structure and extracts the blob URL. It then calls ProcessTask.ExecuteAsync(blobUrl) to begin processing.

Step 3: Payload Loading

ProcessTask reads the blob content from Azure Storage and deserializes the JSON into a TransactionRoot object that will be processed by the rules engine.

Step 4: Rule Set Determination

ProcessTask determines the applicable rule set names from RuleSetNamesConfig, which allows for dynamic loading of tenant-specific or environment-specific rules at runtime.

Step 5: Rules Engine Initialization

A RulesEngine instance is created with ReSettings.CustomTypes populated with RuleHelpers and all static classes from CustomActionHandler.cs, making these methods available to rule expressions.

Step 6: Rule Execution

The engine executes ProcessRulesAsyncDF with the transaction payload as input. Each rule's Expression is evaluated, and if true, the OnSuccess expression is executed.

Step 7: Result Processing

For each rule that evaluates to true, the OnSuccess expression produces either:

  • RuleActionResult object: Contains action type and property updates
  • String identifier: Represents a CustomAction to be invoked
Step 8: Action Handling - RuleActionResult

ProcessTask inspects RuleActionResult objects and applies the appropriate transformation:

  • Update: PropertyUpdater.ApplyValue modifies the existing transaction in place
  • Clone: Original transaction is cloned, property updates are applied to the clone, and the clone is added to the output list
  • CloneWithDerived / CloneWithMultipleDerived: ProcessTask invokes runtime-provided helper logic to compute derived values, then applies them to cloned transactions
Step 9: Custom Action Execution

For CustomAction string results, ProcessTask maps the identifier to a static method in CustomActionHandler using conditional logic or switch statements. The method executes and may update the existing transaction and/or return zero or more cloned transactions.

Step 10: Finalization

After all rules are processed, ProcessTask serializes the final transaction set (original plus any clones or modifications), uploads it to a new output blob in Azure Storage, and enqueues a pointer message for downstream consumers.

Key Flow Characteristics

  • Event-Driven: Processing begins only when Event Grid fires on new blob creation
  • Rule-Based Branching: Each transaction may trigger multiple rules, creating multiple transformation paths
  • Reflection-Based Property Updates: PropertyUpdater.ApplyValue uses reflection to dynamically set properties based on rule definitions
  • Dual Action System: RuleActionResult objects for declarative actions and string identifiers for procedural custom actions
  • Immutable Output: Original blob remains unchanged; all results written to new blob

Rule Model & Action Types

Structure of rules and supported transformations

Rule Structure

Rules are authored as Workflow objects serialized to JSON. Each workflow defines conditions and actions to be applied to transaction payloads.

Key Workflow Fields

WorkflowName

Unique identifier for the rule, used in logging and debugging to trace which rules were evaluated and executed.

Expression

C# boolean expression evaluated against the transaction object. If this expression returns true, the rule's actions are executed. Example: transaction.Amount > 1000 && transaction.Type == "Purchase"

Actions.OnSuccess.Expression

Expression executed when the rule condition is satisfied. This typically invokes RuleHelpers methods that return action descriptors or custom action identifiers.

Supported Action Types

ProcessTask recognizes and handles the following action types returned by rule evaluations:

Update

Purpose: Modify properties on the existing transaction without creating copies.

Use Case: Simple field corrections, status updates, or enrichment of existing records.

Clone

Purpose: Clone the transaction and apply simple property updates to the clone.

Use Case: Creating duplicate transactions with minor variations, such as splitting a transaction across multiple accounts.

CloneWithDerived

Purpose: Clone the transaction and apply derived values obtained via helper logic along with key-value pairs.

Use Case: Creating transactions where values must be calculated from external data sources or complex formulas.

CloneWithMultipleDerived

Purpose: Create multiple cloned transactions, each with different derived values and key-value pairs.

Use Case: Splitting a single transaction into multiple related transactions with computed allocations.

CustomAction

Purpose: Invoke procedural C# methods for complex logic that requires full programmatic control.

Use Case: Complex business logic such as updating an existing transaction and then cloning the updated version, or multi-step transformations.

Component Responsibilities

Key files and their roles in the system

Core Components

AtlasFunction.cs

Responsibility: Azure Event Grid trigger and initial validation

Key Functions:

  • Receives Event Grid notifications for new blob uploads
  • Validates event schema and extracts blob URL
  • Serves as the thin entry point that delegates to ProcessTask
  • Handles initial error conditions and logging
Tasks/ProcessTask.cs

Responsibility: Main orchestrator for the entire processing pipeline

Key Functions:

  • Loads transaction payloads from blob storage
  • Configures and initializes the rules engine with custom types
  • Executes rules and processes their outputs
  • Handles RuleHelpers outputs and invokes runtime-provided helpers
  • Manages action execution for all action types
  • Finalizes output by serializing results, uploading to blob storage, and enqueueing messages
Configuration/RuleSetNameConfig.cs

Responsibility: Dynamic rule set configuration management

Key Functions:

  • Holds the names of rule sets (e.g., for validation rules, business rules)
  • Populated from the service's configuration (environment variables, app settings)
  • Enables loading different rule sets for different tenants or environments at runtime
  • Supports A/B testing and gradual rule rollouts
CustomMethods/CustomActionHandlers.cs

Responsibility: Custom action method declarations and implementations

Key Functions:

  • Declares custom action method signatures exposed to the rules engine
  • Contains static methods invoked by CustomAction identifiers
  • Implements complex business logic that cannot be expressed declaratively
  • Examples: AFOAsiaMethods.ProcessAccrual(), MultiStepTransformations, etc.
Tasks/DataBaseProcessor.cs

Responsibility: Data access abstraction layer

Key Functions:

  • Provides abstraction for database operations during derived value computation
  • Executes SQL queries and returns results to computation logic
  • Handles connection management and query parameterization
  • Supports both synchronous and asynchronous data retrieval
Tasks/Sql/

Responsibility: Embedded SQL resource files

Key Functions:

  • Contains SQL query files referenced by derived value computation logic
  • Loaded as embedded resources at runtime
  • Supports version control and review of SQL logic
  • Enables separation of data access queries from C# code

Component Interaction Pattern

Layered Architecture: The system follows a clear layered pattern where AtlasFunction handles infrastructure concerns, ProcessTask orchestrates business logic, CustomActionHandlers implement domain-specific operations, and DataBaseProcessor manages data access.

Error Handling & Resiliency

Strategies for robust operation

Error Handling Strategy

Fail-Fast at Entry, Graceful Degradation Within: The system validates inputs strictly at the entry point but allows graceful handling of individual rule failures to prevent complete pipeline failures.

Exception Handling Patterns

Entry Point Protection

The ExecuteAsync entry point in ProcessTask is wrapped in comprehensive try/catch blocks. Any unhandled exceptions are:

  • Logged with full context including blob URL, timestamp, and exception details
  • Rethrown to allow Azure Functions runtime retry behavior to activate
  • Tracked in telemetry for monitoring and alerting
Graceful Degradation

Optional data lookups and individual rule evaluations are designed to fail gracefully:

  • Failed data lookups return null or default values rather than throwing exceptions
  • Individual rule failures are logged but do not halt processing of subsequent rules
  • Missing derived values result in null properties rather than pipeline failure
  • Isolated data issues do not cause complete function failure
Validation Gates

Critical validation occurs at key points in the pipeline:

  • Event Validation: Ensures Event Grid payload structure is correct
  • Blob Existence: Verifies blob exists before attempting download
  • JSON Deserialization: Validates payload structure matches TransactionRoot schema
  • Rule Set Loading: Ensures required rule sets are available

Resiliency Mechanisms

Azure Functions Retry

Leverages built-in Azure Functions retry policies for transient failures. Exceptions rethrown from ExecuteAsync trigger automatic retries with exponential backoff.

Idempotency

Processing is designed to be idempotent - reprocessing the same blob produces identical results, allowing safe retries without data corruption.

Timeout Management

Long-running operations include timeout configurations to prevent indefinite hangs. Database queries use command timeouts, and blob operations have retry limits.

Dead Letter Queue

Messages that fail after all retry attempts are routed to a dead letter queue for manual investigation and reprocessing.

Logging and Observability

  • Structured Logging: All log entries include correlation IDs, blob URLs, and relevant context
  • Rule Execution Tracking: Each rule evaluation and action execution is logged with outcomes
  • Performance Metrics: Processing duration, rule evaluation times, and blob operation latencies are tracked
  • Error Categorization: Errors are categorized (transient, configuration, data quality) for appropriate handling

Best Practices

  • Always validate external data before processing
  • Use null-conditional operators and null-coalescing for safe property access
  • Log errors with sufficient context for troubleshooting
  • Design custom actions with exception handling built-in
