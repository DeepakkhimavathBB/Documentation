Executive Summary

This document describes the EnrichmentService, an Azure Function responsible for enriching financial transaction data. The service is triggered by an event containing a pointer to a transaction payload. It fetches the data, performs a series of enrichment steps by querying various database tables and applying business logic, and then passes the enriched data to a downstream service for further processing.

The service is designed as a sequential pipeline, where a central orchestrator (ProcessTask) invokes a series of specialized services to handle different aspects of the enrichment process, such as counterparty details, financial data, investment information, and General Ledger (GL) mapping.

Goals and Scope

Primary Goals

  • To augment raw transaction data with comprehensive details from the system's master data.
  • To apply standardized business logic for determining posting types, GL codes, and other financial attributes.
  • To prepare the transaction data for downstream processing by the EvaluateService.
  • To provide a resilient and idempotent processing pipeline.

Scope

  • This document covers the internal architecture and data flow of the EnrichmentService.
  • It details the responsibilities of each major component and service.
  • It outlines the sequence of enrichment steps performed on each transaction.

High-Level Architecture

  • Trigger: Azure Event Grid (CloudEvent schema). An event is received when a new transaction payload is ready for enrichment. An HTTP trigger is also available for testing and direct invocation.
  • Function App: EnrichmentService (Azure Function).
  • Entry Point: AtlasFunction.cs - A thin wrapper that receives the trigger event and delegates processing to the main orchestrator.
  • Orchestrator: Tasks/ProcessTask.cs - The core of the service. It retrieves the transaction data and orchestrates the entire enrichment pipeline by calling a series of specialized services.
  • Enrichment Services: A set of scoped services, each responsible for a specific part of the enrichment process (e.g., CounterPartyService, FinancialDataService, GlmMappingService).
  • Data Access: DatabaseResultProcessor.cs - A dedicated class for executing SQL queries (embedded as resources) and processing the results.
  • Persistence: The enriched payload is uploaded to a new blob, and a new CloudEvent is enqueued for the next service in the workflow.

End-to-End Flow

  1. An Event Grid message (or an HTTP POST request) triggers the AtlasFunction.
  2. The function validates the input and invokes ProcessTask.ExecuteAsync().
  3. ProcessTask retrieves the transaction payload JSON from the blob URL specified in the event.
  4. The payload, which can contain multiple transactions, is deserialized into a TransactionRoot object.
  5. ProcessTask iterates through each transaction in the TransactionRoot. For each transaction, it executes the following sequence of enrichment steps by calling the corresponding services:
    1. Initial Data Fetch: Fetches baseline data for the transaction using DatabaseResultProcessor.
    2. Transaction Initialization: TransactionInitializer sets up the transaction object.
    3. Financial Data Population: FinancialDataService enriches various financial details.
    4. Posting Type Application: PostingTypeService applies the correct posting strategy.
    5. Counterparty Processing: CounterPartyService enriches counterparty information.
    6. Financial Data Processing: FinancialDataService continues to process financial data.
    7. Local Data Population: TransactionDataService populates local currency information.
    8. Investment Details Processing: InvestmentService enriches investment-specific data.
    9. JE and TGT Grouping: TransactionDataService handles journal entry and transaction grouping.
    10. GLM Mapping: GlmMappingService performs General Ledger mapping.
    11. Status Determination: TransactionInitializer sets the final status of the transaction.
  6. After processing all transactions, the updated TransactionRoot is serialized back into a JSON string.
  7. A new blob containing the enriched JSON is created in storage.
  8. A new CloudEvent is created with a pointer to the new blob and is enqueued for the EvaluateService.
  9. If triggered via HTTP, the enriched JSON is returned directly in the response.

Component Responsibilities and Key Files

EnrichmentService.csproj

Defines the project structure, dependencies (EtonSolutions.ServiceFramework.Runtime, etc.), and embedded SQL files.

AtlasFunction.cs

The function entry point. Handles both Event Grid and HTTP triggers and delegates to ProcessTask.

Program.cs

Sets up the dependency injection container, registering all the services required for the enrichment pipeline.

Tasks/ProcessTask.cs

The main orchestrator. It contains the high-level business logic and calls the various enrichment services in the correct order.

Tasks/DatabaseResultProcessor.cs

Handles all database interactions, executing embedded SQL queries and processing the results.

Tasks/Processors/*.cs

This directory contains the specialized services that are injected into ProcessTask. Each service is responsible for a specific part of the enrichment logic:

  • TransactionInitializer: Initializes and finalizes the transaction object.
  • PostingTypeService: Manages posting type strategies.
  • CounterPartyService: Handles counterparty enrichment.
  • FinancialDataService: Enriches a wide range of financial data.
  • InvestmentService: Processes investment-specific details.
  • GlmMappingService: Manages GLM mapping logic.
  • TransactionDataService: Handles local data, JE, and TGT grouping.
  • BlobUpdater: Updates the JSON payload.
Tasks/Sql/*.sql

Embedded SQL files containing the queries used for data retrieval.

Error Handling and Resiliency

  • The core logic in ProcessTask.ExecuteAsync is wrapped within RunProcessTaskAsync. This is a method provided by the AtlasExecutableTask base class from the EtonSolutions.ServiceFramework.Runtime.
  • This wrapper provides a standardized mechanism for logging, exception handling, and dead-lettering. If an unhandled exception occurs during the process, the base task will catch it, log the details, and move the triggering message to a dead-letter queue for later analysis, preventing infinite retry loops.
  • The service also uses an IIdempotencyService to ensure that the same message is not processed multiple times.
