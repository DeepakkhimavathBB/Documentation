Executive Summary

This document describes the TransactionService, an Azure Function responsible for persisting transaction records to the database. The service is triggered by an event containing a pointer to a transaction payload that has already been enriched. It reads the payload, inserts the transaction records into the appropriate database tables, and then passes a message to a downstream service.

The service’s primary responsibility is persistence. It supports two distinct data flows:

  • Normal Flow — transactions are persisted into the primary dbo schema.
  • Elimination Flow — transactions are persisted into the adj schema.

Goals and Scope

Primary Goals

  • Reliably persist enriched transaction data into the AtlasFive database.
  • Handle both normal and elimination transaction persistence logic.
  • Update related tables with the newly generated transaction key (agt_row_id).
  • Maintain data integrity by updating the in-memory payload with the database-generated key before signaling downstream services.

Scope

  • This document covers the internal architecture and data flow of the TransactionService.
  • It details the persistence logic and handling of different transaction flows.

High-Level Architecture

  • Trigger: Azure Event Grid using the CloudEvent schema. An event is raised when a transaction payload is ready for persistence.
  • Function App: TransactionService (Azure Function).
  • Entry Point: AtlasFunction.cs — minimal entry point that receives the event and delegates processing.
  • Orchestrator: Tasks/ProcessTask.cs — core service logic responsible for retrieving the payload and coordinating database operations.
  • Data Access: Uses the IDatabaseTask interface from EtonSolutions.ServiceFramework.Runtime to execute generic Add and ExecuteQueryAsync operations. No repository pattern is used.
  • State Management: Includes StateUpdateHandler and StateFieldConfig for state-based updates (currently disabled / commented out).
  • Persistence Output: The updated payload (containing generated keys) is re-uploaded to blob storage and a new CloudEvent is enqueued for downstream processing.

End-to-End Flow

  1. An Event Grid CloudEvent triggers AtlasFunction.
  2. AtlasFunction invokes ProcessTask.ExecuteAsync().
  3. ProcessTask retrieves the transaction payload JSON from the blob URL provided in the event.
  4. The payload is deserialized into a TransactionRoot object.
  5. ProcessTask iterates through each transaction in the payload and applies the following logic:
    1. Determine Flow

      The service evaluates whether the transaction belongs to the Elimination Flow using the isElimination flag.

    2. Normal Flow (dbo schema)

      The transaction is inserted into the primary table (e.g., dbo.agt_adv_glbl_trans_primary) using dbTask.Add().

      The database-generated agt_row_id is captured.

      Post-insert update queries are executed to propagate agt_row_id to related tables, such as:

      • gl.gjs_gl_je_smmry
      • dbo.atr_anh_asset_trans
      • dbo.trs_trans_split
    3. Elimination Flow (adj schema)

      The transaction JSON is mapped to an adj_agt_adv_glbl_trans_insert object.

      The record is inserted into the adjustment table (e.g., adj.agt_adv_glbl_trans) using dbTask.Add().

      The generated agt_row_id is captured.

    4. Payload Update

      The in-memory transaction object is updated with the generated agt_row_id.

  6. After all transactions are processed, the updated TransactionRoot is serialized back to JSON.
  7. The updated JSON payload is uploaded back to the original blob, overwriting it.
  8. A new CloudEvent is created and enqueued for the downstream EvaluateService.

Component Responsibilities and Key Files

TransactionService.csproj

Project definition and dependencies, primarily EtonSolutions.ServiceFramework.Runtime. No embedded SQL files.

AtlasFunction.cs

Azure Function entry point that delegates execution to ProcessTask.

Program.cs

Configures dependency injection, registering ProcessTask, StateFieldConfig, and StateUpdateHandler.

Tasks/ProcessTask.cs

Core orchestrator containing persistence logic, flow branching (normal vs elimination), and post-insert update execution.

Tasks/StateUpdateHandler.cs

Helper for state-based field updates (currently not active).

Tasks/StateFieldConfig.cs

Singleton configuration used by StateUpdateHandler.

Error Handling and Resiliency

  • All execution is wrapped by RunProcessTaskAsync, provided by the AtlasExecutableTask base class.
  • This wrapper enforces standardized logging, exception handling, and dead-letter behavior.
  • The service integrates with IIdempotencyService to ensure that duplicate events do not result in duplicate database inserts.
  • Failures are safely captured and surfaced for investigation without causing infinite retry loops.
